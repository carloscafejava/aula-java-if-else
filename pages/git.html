Git & GitHub - Controle de Versão

Domine o sistema de controle de versão mais usado no mundo e trabalhe com confiança em projetos profissionais

1. O Básico: Entendendo Git e GitHub

Se você está começando na programação profissional, vai ouvir muito falar em Git e GitHub. Mas não se preocupe - vamos desvendar esses conceitos de forma clara e prática.

Imagine que você está escrevendo um livro e quer salvar diferentes versões enquanto escreve. Em vez de ter arquivos como "livro_v1", "livro_v2", "livro_FINAL", "livro_FINAL_v2" (isso sim é um pesadelo!), você teria um sistema que rastreia cada mudança. É exatamente isso que Git faz com código!

Dominar Git e GitHub é uma habilidade essencial para qualquer desenvolvedor. Praticamente todas as empresas usam Git para gerenciar código, e GitHub é a plataforma mais popular para hospedar projetos.

2. O que é Git?

2.1. Definição

Git é um sistema de controle de versão distribuído. Isso significa que ele rastreia mudanças no código e permite que você trabalhe com histórico completo de alterações sem depender de um servidor central.

Por que Git é importante?

  • Histórico Completo: Veja cada mudança feita no projeto e por quem
  • Recuperação: Volte a qualquer versão anterior do código
  • Colaboração: Trabalhe com múltiplos desenvolvedores simultaneamente
  • Segurança: Cada desenvolvedor tem uma cópia completa do projeto
  • Branches: Trabalhe em features isoladas sem quebrar o código principal
Conceitos-chave do Git:
bash 
Repository (Repositório)
    └── Local: pasta do projeto no seu computador
    └── Remoto: projeto no GitHub/servidor

Commit (Confirmação)
    └── Snapshot do código em um momento específico
    └── Contém: código, autor, data, mensagem

Branch (Ramificação)
    └── Linha de desenvolvimento separada
    └── main/master: branch principal
    └── feature/*, bugfix/*: outras branches

Working Directory
    └── Seus arquivos no computador

Staging Area
    └── Arquivos preparados para commit

Merge (Mesclagem)
    └── Combinação de duas branches

3. O que é GitHub?

GitHub é uma plataforma web que hospeda repositórios Git. Enquanto Git é o sistema de controle de versão em si, GitHub é a nuvem onde você armazena seus projetos e colabora com outros desenvolvedores.

GitHub oferece:

  • Repositório Remoto: Backup seguro do seu código na nuvem
  • Colaboração: Convide outros para trabalhar no projeto
  • Pull Requests: Revise código antes de mergear
  • Issues: Rastreie bugs e tarefas
  • GitHub Pages: Hospede websites diretamente do GitHub
  • GitHub Actions: Automação de testes e deploy

4. Instalação e Configuração

4.1. Instalar Git

Acesse git-scm.com e baixe a versão para seu sistema operacional.

Verificar Instalação:

bash 
git --version
# Deve mostrar algo como: git version 2.40.0

4.2. Configurar Identidade

Comandos de Configuração:

bash 
git config --global user.name "Seu Nome"
git config --global user.email "seu.email@example.com"

# Verificar configuração
git config --list

Explicação:

  • --global: Aplica para todos os repositórios do seu computador
  • user.name: Seu nome que aparecerá em cada commit
  • user.email: Email que será vinculado aos seus commits

5. Comandos Essenciais do Git

5.1. Inicializar um Repositório

bash 
cd /caminho/do/projeto
git init

# Resultado: Initialized empty Git repository in /caminho/do/projeto/.git/

5.2. Verificar Status

bash 
git status

# Mostra:
# - Arquivos modificados (em vermelho)
# - Arquivos no staging area (em verde)
# - Branch atual

5.3. Adicionar Arquivos (Stage)

bash 
git add Main.java          # Adiciona um arquivo específico
git add .                   # Adiciona TODOS os arquivos modificados
git add *.java             # Adiciona todos os arquivos .java

O que é o Staging Area?

É uma "área de espera" onde você prepara arquivos antes de fazer um commit. Você pode adicionar alguns arquivos, remover outros, revisar mudanças, e só depois fazer o commit quando estiver satisfeito.

5.4. Fazer Commit

bash 
git commit -m "Adiciona função de login"

# Mensagens boas seguem este padrão:
# - Imperativo: "Adiciona" não "Adicionado" ou "Adicionar"
# - Claro e descritivo
# - Explica O QUÊ, não apenas HOW

5.5. Ver Histórico

bash 
git log                    # Mostra histórico completo

git log --oneline          # Resumido (uma linha por commit)

git log -n 5              # Últimos 5 commits

git log --author="Nome"   # Commits de um autor específico

5.6. Trabalhar com Branches

bash 
git branch                           # Lista todas as branches

git branch feature/login            # Cria nova branch

git checkout feature/login          # Muda para a branch

git checkout -b feature/login       # Cria E muda de uma vez

git branch -d feature/login         # Deleta a branch

5.7. Fazer Merge

bash 
git checkout main                    # Volta para branch principal

git merge feature/login             # Mescla a feature na main

# Se houver conflitos, resolva manualmente e depois:
git add .
git commit -m "Resolve conflitos e faz merge de feature/login"

6. Trabalhando com GitHub

6.1. Conectar a um Repositório Remoto

bash 
git remote add origin https://github.com/seu-usuario/seu-repo.git

git remote -v               # Verifica remotos adicionados

6.2. Enviar Código (Push)

bash 
git push -u origin main         # Primeira vez (cria branch remota)

git push                        # Próximas vezes (mais simples)

6.3. Clonar um Repositório

bash 
git clone https://github.com/usuario/repositorio.git

# Cria uma pasta com o projeto e um Git completo

6.4. Atualizar seu Código Local (Pull)

bash 
git pull                    # Atualiza seu código local com mudanças do servidor

7. Workflow Prático (Passo a Passo)

Cenário: Você é contratado para adicionar feature de login

Passo 1: Clonar o projeto do GitHub

git clone https://github.com/empresa/projeto.git
cd projeto

Passo 2: Criar uma branch para sua feature

git checkout -b feature/login

Passo 3: Fazer suas modificações e commits

git add .
git commit -m "Cria tela de login"
git add .
git commit -m "Adiciona validação de senha"

Passo 4: Enviar para GitHub

git push -u origin feature/login

Passo 5: Criar Pull Request no GitHub (pela interface web)

GitHub automaticamente detectará sua branch e oferecerá a opção de criar PR

Passo 6: Outros reviram e aprovam

Depois de aprovado, faz-se merge para a main

9. Erros Comuns e Soluções

❌ Erro 1: "fatal: not a git repository"

bash 
git commit -m "Alguma coisa"
# Erro: fatal: not a git repository

Solução: Você esqueceu de fazer git init primeiro!

❌ Erro 2: Merge Conflicts

bash 
git merge feature/login
# Erro: CONFLICT (content merge conflict)

Solução: Abra os arquivos em conflito, escolha qual versão manter, e faça commit!

✅ Boas Práticas:

  • Sempre faça git pull antes de começar a trabalhar
  • Commits pequenos e frequentes (não um gigante no final)
  • Mensagens de commit claras e em português ou inglês consistente
  • Nunca trabalhe diretamente na main, use branches
  • Revise código antes de mergear (code review)
  • Use .gitignore para ignorar arquivos desnecessários

10. Exercícios Práticos

📝 Exercício 1: Setup Inicial

Crie um novo repositório local, configure seu nome/email, crie um arquivo Main.java com um "Hello World" e faça o primeiro commit.

📝 Exercício 2: Branches e Merge

Crie uma branch chamada "feature/calcular", modifique um arquivo, faça commit, volta para main e faz merge.

📝 Exercício 3: GitHub

Crie uma conta no GitHub, crie um novo repositório público e faça push do seu projeto local.

📝 Exercício 4: Clone e Colaboração

Clone um repositório público do GitHub, faça uma modificação local e tente fazer push (você não conseguirá - isso é proposital! Fork depois e tente novamente).

📝 Exercício 5: Histórico

Faça pelo menos 5 commits com mensagens diferentes. Use git log para visualizar o histórico completo e git log --oneline para ver resumido.

🎯 Conclusão

Parabéns! Você agora tem os fundamentos de Git e GitHub. Esses conhecimentos serão essenciais em sua carreira como desenvolvedor. A prática leva à perfeição - quanto mais você usar Git, mais natural ficará.

Próximos passos: Comece a usar Git em seus projetos pessoais, contribua em projetos open source no GitHub, e explore recursos avançados como rebase, cherry-pick e stash.
